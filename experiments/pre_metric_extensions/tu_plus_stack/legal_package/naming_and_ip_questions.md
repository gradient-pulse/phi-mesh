# T-Protocol — Naming and IP Questions

## Purpose

This file captures the current naming, protection, disclosure, and IP questions surrounding T-Protocol.

It is not yet a legal opinion and not a finished licensing instrument.

Its purpose is to:
- identify the main legal and commercial protection issues early
- prepare for later work with IP counsel
- reduce accidental disclosure mistakes
- clarify what needs decision before broader circulation

---

## 1. Naming questions

### 1.1 Primary public name
Current leading name:
- **T-Protocol**

Reasons:
- short
- memorable
- commercially usable
- broad enough for infrastructure framing
- less bulky than “Triad Protocol”

Open question:
- is **T-Protocol** protectable enough as a name, or is it too generic on its own?

### 1.2 Secondary / descriptive name
Current descriptive backup:
- **Triad Protocol**
- **Triadic Coordination Protocol**
- **T-Protocol for LLM Coordination**

Open question:
- should there be a formal pairing such as:
  - **T-Protocol™**
  - subtitle: **Triadic Coordination Protocol for LLM Systems**

### 1.3 Naming strategy question
Open question:
- should one name be used for public category formation and another for formal legal registration?

### 1.4 Geographic question
Open question:
- in which jurisdictions should naming protection first be considered?
  - Benelux
  - EU
  - UK
  - US
  - other

## 3. Protocol disclosure boundaries

### 3.1 Public layer
Current public-safe material appears to include:
- short conversation note
- introductory architecture brief
- public diagram
- high-level category framing

Open question:
- is any currently public material already too revealing from an IP perspective?

### 3.2 Protected review layer
Protected material appears to include:
- role specifications
- bounded-flow rules
- feedback discipline
- operating modes in practical detail
- compliance logic
- evidence map
- selected back-package materials

Open question:
- what exact threshold should trigger NDA or confidentiality protection before sharing?

### 3.3 Licensed layer
Licensed material appears to include:
- full implementation-facing package
- near-operational instructions
- implementation guidance
- deployment logic
- faithful-use conditions

Open question:
- how should “implementation-grade substance” be defined clearly enough to enforce?

Current evidence already includes:
- repository history

- observations, laws, metrics, failure taxonomy, assessments

## 5. Public vs protected vs licensed-only

### 5.1 Current need
A defensible disclosure map is needed so that:
- public conversation does not unintentionally disclose reconstructable protocol substance
- protected review is disciplined
- licensed use is clearly separated from review access

### 5.2 Open questions
- which current files should be reclassified before wider circulation?
- does any front-package material already belong in protected review only?
- should some role or flow materials be split into:
  - summary version
  - implementation version

### 5.3 Practical test
Working rule:

> If a capable team could reconstruct a meaningful version of T-Protocol from the document, it should not be treated as ordinary public material.

Open question:
- does this rule need refinement into a more operational internal checklist?

## 6. Faithful implementation question

### 6.1 Core issue
A legal and commercial definition is needed for what counts as a **faithful T-Protocol implementation**.

### 6.2 Candidate elements
Likely required elements include:
- triadic role structure
- bounded information flow
- recursive feedback discipline
- mode-governed operation
- preservation of role distinction
- no material flattening into generic prompt chaining

### 6.3 Open question
- what minimum set of elements must be present before use of the name “T-Protocol” is allowed?

---

## 7. Modification and derivative-use questions

### 7.1 Core issue
If a licensee modifies the protocol, when does it remain:
- a valid T-Protocol implementation
versus
- a derivative inspired by T-Protocol
versus
- something that should not use the name at all?

### 7.2 Open questions
- what modifications should be allowed?
- what modifications should require approval?
- should there be a distinction between:
  - internal tuning
  - deployment adaptation
  - structural alteration of the protocol grammar

## 9. Copyright / trade secret / trademark mix

Protection may not come from one mechanism alone.

Possible protection mix:
- copyright in the documents
- trademark in the name
- trade-secret treatment of implementation-grade materials
- contract-based confidentiality and licensing restrictions
- timestamped origin trail

### 9.2 Open question
- what is the right practical mix for T-Protocol at this stage?
