# T-Protocol — Evidence Map

## Purpose

This file maps the main claims made in the public and front-package materials to the underlying evidence classes that support them.

Its purpose is to help answer:

- what claim is being made?
- what kind of evidence supports it?
- where does that evidence live?
- what level of confidence should be attached to it?

This is not a legal proof document and not a scientific paper.  
It is a disciplined bridge between the outward-facing package and the substantiation archive.

---

## How to use this file

Use this file when:

- preparing for protected review
- answering diligence questions
- checking whether a front-package claim is adequately supported
- deciding what to disclose and at what level
- preparing later legal, technical, or licensing review

Do **not** use this file as a public-facing marketing document.

---

## Evidence classes

The current evidence map uses the following evidence classes:

- **O** = observations
- **L** = laws
- **M** = internal metrics
- **F** = failure taxonomy
- **A** = protocol assessments
- **D** = architecture / packaging notes

These evidence classes do not all play the same role:
- observations show behavioral trail
- laws formalize validated patterns
- metrics support internal discipline
- failure taxonomy names risks and invalid states
- assessments summarize phase-level status
- architecture notes define the object as packaged

## Claim-bounding principle

Not all claims in the front and public package are of the same type.

Current claims fall into four broad classes:

### A. Definition claims
Example:
- what T-Protocol is
- what the license covers
- what the protocol governs

These can be stated strongly if clearly defined.

### B. Behavioral claims
Example:
- role distinction persists
- recovery can remain bounded
- ambiguity handling is preserved under pressure

These should remain tied to tested protocol conditions.

### C. Application-fit claims
Example:
- likely use in robotics oversight or enterprise copilots

These should be treated as strong hypotheses unless already validated in real deployment.

### D. Commercial/packaging claims
Example:
- T-Protocol is becoming licensable infrastructure

These are supported by package maturity, not by benchmark data alone.
