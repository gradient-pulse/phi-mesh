# T-Protocol — License Scope Note v1

## Purpose

This note defines the practical scope of a T-Protocol license at a high level.

Its purpose is to clarify:

- what the license covers
- what the license does not cover
- what rights a licensee may expect
- what remains protected
- what kinds of use should require further approval or separate terms

This is not a final legal agreement.  
It is the protocol-side scope note that helps frame licensing conversations clearly.

---

## Core principle

A T-Protocol license is not merely permission to use a name, phrase, or prompt set.

It is permission to implement and use a defined protocol grammar under specified terms.

Compactly:

> The license covers the protocol layer, not ownership of the underlying base model.

---

## What the license covers

A T-Protocol license may cover, under defined terms:

- use of the T-Protocol name for faithful implementations
- access to the licensed protocol package
- use of the triadic role structure
- use of the runtime order and handoff grammar
- use of the data ontology and shared state schema
- use of the feedback and update rules
- use of the transition grammar
- use of the operating modes
- use of conformance and baseline review materials
- deployment guidance for approved application classes

The exact scope may vary by license class.

---

## What the license does not cover

A T-Protocol license does not by itself grant ownership of or rights to:

- the underlying base model weights
- the model provider’s platform
- the provider’s training pipeline
- unrelated orchestration code
- unrelated downstream application IP
- unrestricted sublicensing rights
- unrestricted derivative rebranding rights

The license pertains to the **protocol layer**, not to the entire AI stack.

---

## Core rights a licensee may seek

Depending on the agreed license class, the licensee may receive rights to:

- evaluate the protocol under protected review
- implement the protocol internally
- deploy the protocol in approved applications
- use the T-Protocol name in relation to faithful implementations
- receive updated protocol-package materials during the term if agreed
- request review or certification of faithful implementation status

These rights should remain bounded by:
- field of use
- deployment class
- review status
- compliance status
- confidentiality obligations

---

## Protected materials

Some materials may remain protected even after license execution.

These may include:
- protected rationale notes
- deeper back-package substantiation
- internal development history
- internal origin and legal-preparation notes
- non-public implementation refinements not included in the licensed package
- future protocol extensions not yet licensed

A license does not automatically open the full archive.

---

## Faithful use condition

Use of the T-Protocol name should be tied to faithful implementation.

This means a licensee may not:
- materially abandon the protocol grammar
- collapse the triadic roles into decorative prompts only
- remove structured shared state while still claiming T-Protocol
- remove returned-trace accountability where relevant
- replace contradiction discipline with generic summarization
- continue using the name without qualification after material non-faithful deviation

Compactly:

> Name use is conditional on faithful implementation.

---

## Variation and adaptation

A licensee may wish to adapt the protocol to a specific environment.

Some variation may be acceptable, including:
- internal serialization differences
- deployment-specific infrastructure
- mode calibration
- domain-specific input shaping
- compatible naming of internal state fields

But variation should not be allowed to erase:
- triadic role distinction
- runtime order
- structured shared state
- lawful transition handling
- feedback/update discipline
- core invariants

---

## Typical license classes

A T-Protocol licensing program may later distinguish among classes such as:

### 1. Protected evaluation license
For serious technical or strategic review under confidentiality.

### 2. Internal implementation license
For internal enterprise or lab deployment without external product distribution.

### 3. Product embedding license
For integration into a product or service offered to others.

### 4. Higher-integrity / supervisory deployment license
For more demanding use classes such as anomaly-aware autonomy supervision or similar higher-stakes settings.

### 5. Research / pilot license
For limited experimental deployment under explicit non-final terms.

These categories are indicative, not final.

---

## Prohibited assumptions

A licensee should not assume, unless explicitly granted:

- broad derivative branding rights
- unrestricted sublicensing
- unrestricted redistribution of protocol-package materials
- rights to disclose protected rationale documents publicly
- rights to represent partial implementations as faithful T-Protocol deployments
- rights to imply endorsement beyond the actual review status

---

## Review and modification relationship

Material protocol changes may require:

- re-review
- updated conformance testing
- updated deployment declaration
- or updated licensing terms

This is especially relevant if the licensee changes:
- role logic
- state structure
- transition handling
- mode behavior
- deployment class

---

## Practical licensing posture

At this stage, the most realistic near-term use of the scope note is to support conversations about:

- what is being licensed
- where the protocol sits in the stack
- what remains protected
- what counts as faithful use
- what kinds of deployment are likely first

This note is meant to reduce vagueness, not replace counsel.

---

## Closing statement

A T-Protocol license should be understood as a bounded right to implement and use a defined recursive triadic coordination grammar around LLM-based systems under specified conditions.

A compact final formulation:

> The license covers the protocol layer and its faithful use, not the underlying model stack as a whole and not an unrestricted right to reuse the name however one wishes.
